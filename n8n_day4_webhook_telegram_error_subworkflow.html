Webhook 방식 (현재)

외부에서 N8N에 알림

이벤트가 생기는 순간 외부 서비스가 먼저 N8N에 데이터를 보냅니다.
결제 완료 → 즉시 N8N 실행 → 즉시 처리.
장점: 진짜 실시간, 즉각 반응

Webhook Trigger 노드 설정
// ─── Webhook Trigger 노드 ───
HTTP Method:  POST          // 또는 GET (데이터 전송엔 POST 권장)
Path:         my-webhook  // URL 뒤에 붙는 경로 (자유롭게 설정)
Respond:      Immediately   // 요청받자마자 200 OK 응답
Response Code: 200

// ─── 생성된 Webhook URL (2가지) ───
// Test URL: 테스트할 때만 사용 (노드 열려있는 동안만 활성화)
Test:  http://localhost:5678/webhook-test/my-webhook

// Production URL: Active ON 후 항상 열려있는 URL
Prod:  http://localhost:5678/webhook/my-webhook

// ─── 외부 접근이 필요하면 ngrok 사용 ───
// ngrok http 5678 → https://xxxx.ngrok.io/webhook/my-webhook
⚠️ Test URL vs Production URL 차이: Test URL은 N8N에서 Webhook 노드를 클릭해서 열어둔 상태에서만 동작합니다. 실제 운영(Active ON)에서는 반드시 Production URL을 사용해야 합니다. 외부 서비스(Postman, 결제 시스템 등)에 등록할 때는 Production URL을 씁니다.
Webhook으로 받은 데이터 접근법
// POST 요청으로 이런 데이터가 들어왔다고 가정:
{
  "이름": "홍길동",
  "이메일": "hong@test.com",
  "금액": 50000
}

// ─── N8N 표현식으로 접근 ───
{{ $json.body.이름 }}      // POST Body 데이터
{{ $json.body.금액 }}
{{ $json.headers['x-api-key'] }}  // HTTP 헤더
{{ $json.query.page }}     // URL 쿼리스트링 (?page=2)

// GET 요청 예시: /webhook/search?keyword=사과&limit=10
{{ $json.query.keyword }}  // "사과"
{{ $json.query.limit }}    // "10"
방식데이터 위치N8N 접근법언제 쓰나요?
POST 요청 Body (JSON) $json.body.필드명 폼 전송, 결제 알림, 챗봇 응답
GET URL Query String $json.query.파라미터 검색, 조회, 페이지 요청
🧪 실습 14 — Postman/curl로 Webhook 호출하기
N8N Webhook을 외부에서 호출하는 방법입니다. Postman이 없으면 curl로 터미널에서 바로 테스트할 수 있어요.
curl 명령어 — 터미널에서 바로 테스트
# POST 요청 — JSON 데이터 전송
curl -X POST http://localhost:5678/webhook-test/my-webhook \
  -H "Content-Type: application/json" \
  -d '{"이름":"홍길동","이메일":"hong@test.com","금액":50000}'

# GET 요청 — Query Parameter
curl http://localhost:5678/webhook-test/my-webhook?keyword=사과&limit=5

# API Key 헤더 포함
curl -X POST http://localhost:5678/webhook-test/my-webhook \
  -H "Content-Type: application/json" \
  -H "X-API-Key: my-secret-key-123" \
  -d '{"주문번호":"ORD-001","상태":"결제완료"}'

# Windows PowerShell
Invoke-WebRequest -Uri http://localhost:5678/webhook-test/my-webhook `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"이름":"홍길동"}'

Telegram Bot 자동화

BotFather로 봇 생성부터 양방향 대화까지
1

BotFather로 봇 생성

텔레그램에서 @BotFather 검색 → /newbot 입력 → 봇 이름 입력 → 봇 아이디 입력 (영문, _bot으로 끝나야 함) → API Token 발급

2

N8N Credential 등록

N8N → Credentials → "+ Add" → Telegram API → Access Token에 BotFather에서 받은 토큰 붙여넣기 → Save

3

Chat ID 확인

봇에게 아무 메시지 전송 → 브라우저에서 api.telegram.org/bot{TOKEN}/getUpdates 접속 → message.chat.id 값 복사

4

N8N Telegram 노드 연결

Telegram 노드 → Credential 선택 → Chat ID 입력 → 메시지 작성 → Active ON

Telegram Trigger — 메시지 받으면 자동 응답
✈️ Telegram Trigger 노드: 사용자가 봇에게 메시지를 보내면 N8N이 자동으로 감지하고 워크플로우를 실행합니다. IF 노드와 조합해서 명령어별 다른 응답을 만들 수 있어요.
Telegram Trigger → IF → 명령어별 응답
// ─── Telegram Trigger 노드 ───
Updates: message           // 메시지 수신 감지

// 수신 데이터 접근
{{ $json.message.text }}     // 받은 메시지 내용
{{ $json.message.chat.id }}   // 보낸 사람 Chat ID
{{ $json.message.from.first_name }}  // 이름

// ─── IF 노드로 명령어 분기 ───
조건: {{ $json.message.text }} = "/환율"
  → TRUE:  환율 API 호출 후 응답
  → FALSE: "명령어를 확인하세요.\n/환율 - 오늘 환율\n/날씨 - 날씨 정보"

// ─── Switch 노드로 여러 명령어 처리 ───
Rule 1: text = "/환율"   → 환율 조회 플로우
Rule 2: text = "/날씨"   → 날씨 조회 플로우
Rule 3: text = "/도움말" → 도움말 메시지
Fallback                  → "알 수 없는 명령어입니다."

// ① Telegram Trigger: "/보고서" 수신
// ② Google Sheets: Get Rows (최근 7일 데이터)
// ③ Code 노드: 통계 계산

const rows = items;
const rates = rows.map(i => Number(i.json.USD_KRW));
const temps = rows.map(i => Number(i.json.기온));

const avgRate = Math.round(rates.reduce((a,b)=>a+b,0)/rates.length);
const maxRate = Math.max(...rates);
const minRate = Math.min(...rates);
const avgTemp = (temps.reduce((a,b)=>a+b,0)/temps.length).toFixed(1);

const chatId = $("Telegram Trigger").item.json.message.chat.id;
const userName = $("Telegram Trigger").item.json.message.from.first_name;

return [{
  json: {
    chatId,

💱 *USD 환율 (최근 ${rows.length}일)*

• 최고: ${maxRate.toLocaleString()}원

_Google Sheets 자동 분석 완료_`
  }
}];

에러 처리 & 안정적인 워크플로우

실패해도 자동 복구되는 견고한 자동화 설계
에러 처리

3가지 에러 처리 패턴

Continue on Fail — 실패해도 다음 노드 계속
Retry on Fail — 자동 재시도 N회
Error Trigger — 실패 시 별도 워크플로우 실행

안정성 높이기

4가지 안정성 강화 도구

Wait 노드 — Rate Limit 대응 딜레이
Variables 노드 — 전역 변수 관리
Note 노드 — 워크플로우 문서화
실행 시간 최적화 — 불필요한 노드 줄이기

Continue on Fail + Try/Catch 패턴
// ─── Continue on Fail 설정 ───
// HTTP Request 노드 우클릭 → Settings → "Continue On Error" ON
// 실패 시에도 다음 노드로 이동 (에러 정보 포함)

// ─── IF 노드로 성공/실패 분기 ───
조건: {{ $json.error }} Exists (에러 있으면 TRUE)
  TRUE:  에러 처리 경로 (로그 기록, 알림)
  FALSE: 정상 처리 경로

// ─── Code 노드에서 Try/Catch 패턴 ───
for (const item of items) {
  try {
    // 실패할 수 있는 작업
    item.json.결과 = JSON.parse(item.json.rawData);
    item.json.성공 = true;
  } catch (e) {
    item.json.성공 = false;
    item.json.에러 = e.message;
    // 에러가 나도 다른 아이템 계속 처리
  }
}
return items;

Sub-Workflow & 워크플로우 재사용

모듈화된 자동화로 유지보수 효율 극대화
Sub-Workflow란?

워크플로우 안에서 다른 워크플로우 호출

공통으로 쓰이는 로직(환율 계산, 알림 전송 등)을 별도 워크플로우로 만들어두고 여러 곳에서 호출합니다. 한 번 수정하면 모든 곳에 반영!

언제 쓰나요?

반복되는 로직 분리

✅ 텔레그램 알림 전송 (5개 워크플로우에서 공통 사용)
✅ 환율 조회 & 계산 로직
✅ Google Sheets 기록 공통 모듈
✅ 에러 알림 공통 처리
